# 写作工作流说明书

这个文件是给 AI（Claudian）看的。用户会用中文聊天，你按下面的规则操作他的 Obsidian vault。**你直接读写文件，不需要调用任何外部命令或脚本。**

## 三条规则

1. **一篇文章一个文件夹**：所有东西都在 `articles/<文章名>/` 下面。文章名用中文就行。
2. **source 只读，working 可写**：初稿存一份到 `source.md`，永远不动；所有修改都在 `working.md` 上做（用户自己改、让你改，都可以）。
3. **只往前走**：想法 → 初稿 → 工作稿 → 定稿，单向流动。想改已定稿？回到 working.md 改，重新定稿会覆盖 final.md。

## 三个动作

### 导入工作流
用户说："导入工作流，文章名叫 xxx"、"放进工作流"——意图类似即可。

你做：
- 找到对应的 idea 笔记：用户指定了哪条就用哪条；没说就用最近编辑的 `idea/*.md`；判断不了就问用户（**不要猜**）
- 基于 idea 笔记里的想法和要点，**扩写成一篇完整文章**——这是关键，用户期望 source.md 里就是一篇能读的文章而不是原始想法
- 建好目录 `articles/xxx/draft/`
- 把扩写出的文章**同时**写入 `articles/xxx/draft/source.md` 和 `articles/xxx/draft/working.md`（两份内容完全一致）
- 用一句话回确认

### 修改 working.md
用户说："改开头"、"这段太长"、"润色一下"、"加个总结"、"写短一点"——任何跟"改"相关的意图。

你做：
- **改之前先读一次 `memory/我的改稿习惯.md`**（如果不为空），按里面的偏好去改
- 只编辑 `articles/<当前文章>/draft/working.md`
- 不要动 source.md

### 定稿
用户说："定稿"、"就这样吧"、"可以了"、"我满意了"——意图类似即可。

你做：
- 建好目录 `articles/<当前文章>/final/`
- 把 `working.md` 的内容复制到 `articles/<当前文章>/final/final.md`
- **生成封面提示词**（见下面「生成封面提示词」），写入 `articles/<当前文章>/final/cover-prompt.md`
- **顺便学习这次的改稿习惯**（见下面「自动学习」）
- 回复用户：确认定稿 + 封面提示词已生成 + 这次学到了什么

## 生成封面提示词

定稿时一起生成，写入 `articles/<文章名>/final/cover-prompt.md`。基于 `final.md` 的主题和情绪，用下面的模板产出：

~~~markdown
# 封面提示词 · 《<文章名>》

## 图片概念

<一两句中文描述图片的核心意象——读者一眼看到封面要感受到什么。抓情绪和象征，不要剧透正文>

## 英文 prompt（Midjourney / DALL-E / Stable Diffusion）

<主体 + 风格 + 构图 + 光线氛围 + 画幅比例，逗号分隔，默认 --ar 16:9>

## 中文 prompt（即梦 / 可灵 / 文心一格）

<具体、有画面感的中文描述，长度和英文 prompt 对等，默认 16:9>
~~~

如果 `memory/我的改稿习惯.md` 里记了封面/画风的偏好（比如"喜欢极简插画"、"别用赛博朋克"），**按偏好来**。

## 自动学习改稿习惯

定稿时，对比 `source.md` 和 `working.md` 的差异，提炼用户的写作偏好，追加到 `memory/我的改稿习惯.md`。

### 追加格式

把观察到的规律追加到文件末尾，每次定稿一小段，形如：

~~~
## YYYY-MM-DD · 来自《文章名》定稿

🤖 [观察] 具体规律的一句话概括
   实例：在这一篇里，你把「X」改成了「Y」

🤖 [观察] 另一条规律
   实例：……
~~~

### 特殊情况处理

- **差异太小**（只改了几个字、没明显规律）：**跳过，不追加**。定稿回复里说明"这次改动太少，没学到新习惯"。
- **和用户手写的偏好冲突**：仍然追加，标成冲突条目——

~~~
🤖 [冲突] 我的观察和你之前写的不一致
   我观察到：<具体行为>
   你手写过（YYYY-MM-DD）：<原话>
   以你手写为准。如果想更新，跟我说一声就好。
~~~

- **和自己之前的观察不同**（比如之前学到「偏好短句」，这次观察到「长段抒情」）：两条都留，按时间追加。以后读的时候，**根据当前文章题材自己判断**该参考哪条，不要强制合并或覆盖。

### 定稿时告诉用户

回复结构示例：

> 定稿好了 ✅
> 这次从你的改动里学到 2 条新习惯，已加到《我的改稿习惯》：
> - 短句节奏（你把 3 个长句拆成了 7 个短句）
> - 删掉感叹号
>
> 如果有不准的，直接在 `memory/我的改稿习惯.md` 里删掉就行。

## 手动触发学习

用户说："学一下我的偏好"、"记一下我刚才改的"、"把这次改稿习惯记下来"、"刚才改了什么你学一下"——**任何意图是"把这次改动沉淀下来"的话**都触发。

你做：和「自动学习」同样的流程——对比 source vs working，追加到 memory，告诉用户学到了什么。

## 删除偏好

用户说："忘掉关于 X 的偏好"、"把感叹号那条删了"——**任何意图是让你删某条改稿习惯的话**。

你做：直接编辑 `memory/我的改稿习惯.md`，删掉对应条目。

（用户也可能自己打开 `memory/我的改稿习惯.md` 手动删——那是 OK 的，**不要补回来**。手写优先于你的观察。）

## 看状态

用户问"这篇现在什么状态？"、"走到哪了？"——不是动作，是**判断**。看文件：

- 只在 `idea/` 有对应笔记 → **想法池**
- 有 `draft/working.md` → **正在写/改**
- 有 `final/final.md` → **已定稿**

用中文告诉用户，不需要专门的状态文件。

## 记录想法

用户说："我有个想法：xxx"、"记一下：xxx"、"突然想到：xxx"——**任何意图是"把一个想法先记下来"的话**。

你做：先判断这是**新想法**还是**对刚才那条想法的补充**——

- **全新想法**（和上下文没关联）：在 `idea/` 里新建一个 md 文件（文件名用关键词，中文即可），把内容写进去
- **补充已有想法**（用户说"再补充几个点"、"还有一个点"、"加一条"，或明显是在接着上一条想法聊）：**追加**到对应的 idea 笔记里，不要建新文件——让同一条笔记长大

判断不了是新增还是补充？**问用户，不要猜。**

## 小心事项

- 判断不了用户说的是哪篇文章？**问用户，不要猜。**
- 目标文件/文件夹已存在？**不要直接覆盖，问用户。**
- 手写偏好永远优先于你观察到的。
